# 機能案：復習日リマインダー通知

> ⚠️ **エージェントへ：** この機能は構想段階です。ユーザーが明示的に実装を指示するまで実装しないでください。

**目的：** `next_review_at` が当日になった間違いノートがあることをアプリ外（メール等）で知らせ、「アプリを開かないから復習しない」を防ぐ。

---

## 前提

- **認証（Phase 3）完了後に着手する。** 現在は seed ユーザー1人のみでメールアドレスも持っておらず、通知先が存在しない
- 定期実行の基盤（毎朝◯時にチェックする仕組み）が新たに必要になる。他の機能案と違い、リクエスト駆動の FastAPI だけでは完結しない

---

## 概要（UX）

- 毎朝決まった時刻（例: 7:00、ユーザー設定可）に「今日の復習が◯件あります」というメールを送る。対象は `status="active"` かつ `next_review_at <= today` の件数（既存 `/mistake-notes/today` と同じ条件）
- 0件の日は送らない。通知の ON/OFF と時刻は設定画面で変更できる
- 将来的には PWA のプッシュ通知に拡張（メールより即時性が高いが、Service Worker・購読管理が必要なので第2段階）

---

## API/DB 変更点

| 種別 | 変更 |
|------|------|
| DB | `users` に `email`（Phase 3 で追加予定）、`notify_enabled` (Boolean), `notify_time` (Time) を追加 |
| API | `GET/PUT /v1/me/notification-settings` |
| 定期実行 | ローカル/ECS: APScheduler をアプリ内で持つ or 別プロセスの cron スクリプト。AWS 移行後: EventBridge Scheduler → Lambda が自然 |
| メール送信 | ローカル開発: MailHog 等でダミー送信 / 本番: Amazon SES |
| フロント | 設定画面（通知 ON/OFF・時刻） |

---

## 実装ステップ（Phase 3 完了後)

1. `users` に通知設定カラム追加＋設定 API
2. 「通知対象ユーザーと件数を集めてメールを送る」処理を単体の関数/スクリプトとして実装（定期実行基盤と分離してテスト可能に）
3. ローカルは cron または APScheduler で毎分チェック→該当時刻のユーザーに送信
4. フロントの設定画面
5. AWS 移行時に EventBridge + SES に載せ替え

---

## 検討事項・トレードオフ

- 差別化度は低い（どの学習アプリにもある）が、継続率への効果は大きい定番機能。優先度はロードマップ後半で十分
- タイムゾーン: 当面は JST 固定でよい
- メールの到達性（SPF/DKIM）は SES 利用時に設定。ローカル検証段階では気にしない
- プッシュ通知（PWA）はメール版の利用状況を見てから判断する
