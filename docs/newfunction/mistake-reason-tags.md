# 機能案：間違い原因タグ（弱点の構造化）

> ⚠️ **エージェントへ：** この機能は構想段階です。ユーザーが明示的に実装を指示するまで実装しないでください。

**目的：** 「なぜ間違えたか」をフリーテキスト（memo）だけでなく**構造化されたタグ**として記録し、「自分はどういうタイプのミスが多いのか」を可視化できるようにする。このアプリ最大の差別化候補。

---

## 概要（UX）

- 登録画面・復習画面で、間違えた理由のテキスト欄の上に**原因タグのチップ**を置き、タップで選択する（例: 計算ミス / 知識不足 / 読み間違い / 解法選択ミス / 時間切れ / ケアレスミス）
- タグ＝分類、memo＝詳細。両方書いてもタグだけでもよい（クイック保存との相性◎：理由の文章化は面倒でもタグ1タップならできる）
- 蓄積されると「あなたの間違いの 42% は計算ミス」「英語は知識不足、数学は読み間違いが多い」のような弱点分析ができる（→ [stats-dashboard.md](./stats-dashboard.md)）

---

## 設計方針：まず単一タグの enum から始める（推奨）

| 案 | 内容 | 評価 |
|----|------|------|
| A: 単一タグ（推奨） | `mistake_notes.reason_tag` にプリセット enum を1つ持たせる | 変更最小・集計が単純。まずこれで価値検証 |
| B: 多対多＋ユーザー定義 | `mistake_tags` ＋ `mistake_note_tags` 中間テーブル、ユーザーが自由にタグ作成 | 柔軟だがテーブル2つ・UI・集計すべて重い。A で不足を感じてから |

プリセットタグ案（enum 値）: `calculation`（計算ミス）/ `knowledge`（知識不足）/ `misread`（読み間違い）/ `approach`（解法選択ミス）/ `careless`（ケアレスミス）/ `time`（時間切れ）/ `other`（その他）

---

## API/DB 変更点（A案）

| 種別 | 変更 |
|------|------|
| DB | `mistake_notes` に `reason_tag` カラム追加（Enum, nullable）。Alembic マイグレーション |
| スキーマ | `QuestionCreate` / `MistakeNoteUpdate` / note 系レスポンスに `reason_tag` を追加 |
| API | 既存の `POST /v1/questions`・`PUT /v1/mistake-notes/{id}` で受け付ける。`GET /v1/mistake-notes` に `reason_tag` フィルタ追加（任意） |
| フロント | 登録・復習・間違い一覧・詳細にタグチップ UI。一覧ではタグをバッジ表示 |

---

## 実装ステップ

1. `reason_tag` カラム追加のマイグレーション＋モデル/スキーマ更新
2. questions / mistake_notes ルーターで受け渡し（`_build_response` 系にも追加）
3. `openapi.json` 再生成 → `npm run generate`
4. 登録画面にタグチップ、間違い一覧にバッジ表示
5. （後続）stats-dashboard でタグ別集計

---

## 検討事項・トレードオフ

- タグ名の文言は学生に伝わる言葉にする（「解法選択ミス」→「やり方が思いつかなかった」等、UI 表示名と enum 値を分離する）
- 教科によって適切なタグが違う（数学: 計算ミス / 英語: 単語不足）→ まずは共通プリセットで開始し、科目別カスタムは B 案に進むときの検討事項
- 既存データは `reason_tag = null` のまま（nullable なので移行不要）。「未分類」として集計から除くか「その他」に含めるかは統計側で決める
