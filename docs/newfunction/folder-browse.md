# 機能案：科目フォルダ・ブラウズ（問題の階層一覧）

> ⚠️ **エージェントへ：** この機能は構想段階です。ユーザーが明示的に実装を指示するまで実装しないでください。

**目的：** 過去に登録した問題を、ファイルアプリのように「科目フォルダ → 単元フォルダ → 問題」とたどって振り返れるようにする。現状は問題をダッシュボード（今日の復習）か苦手問題ページからしか見られず、「あの科目のこの単元の問題を見返したい」という科目・単元起点の振り返り導線がないのを補う。

---

## 概要（UX）

- ナビに「フォルダ」ページ（例: `/browse`）を追加する
- **ルート** = 自分の科目フォルダ一覧（プリセットや手動で登録した `subjects`）
- 科目フォルダを開くと:
  - **単元がある科目** → 単元フォルダの一覧を表示。単元フォルダを開くとその単元の問題一覧
  - **単元がない科目** → 問題一覧を直接表示
- **単元を持つ科目に「単元なし」で登録された問題**（`unit_id = null`）は、科目直下に「未分類」フォルダ（または一覧の一区画）としてまとめて表示する
- 各フォルダには**問題の件数バッジ**を出す（例: 数学Ⅰ 12 / 二次関数 5）
- パンくず（ホーム > 数学Ⅰ > 二次関数）で階層を戻れる
- 問題をタップすると既存の問題詳細へ遷移する（そこから復習・編集は既存機能に接続）

> ファイラ風の「探して振り返る」導線であり、復習キュー（今日の復習）とは役割が別。今日やるべき復習＝ダッシュボード、任意の範囲を見返す＝このフォルダ、という住み分けにする。

---

## 設計方針：既存APIでほぼ実現できる

この機能の中身（科目・単元・問題の階層）は、すでにある読み取りAPIの組み合わせでほとんど賄える。**新規テーブルは不要**。

| 階層 | 使うAPI（既存） |
|------|----------------|
| ルート（科目一覧） | `GET /v1/subjects` |
| 科目内の単元一覧 | `GET /v1/subjects/{id}/units` |
| 単元の問題一覧 | `GET /v1/questions?subject_id=…&unit_id=…` |
| 単元なし科目の問題一覧 | `GET /v1/questions?subject_id=…` |

そのため実装は**フロント側の画面追加が主**で、バックエンドは下記の小さな隙間だけを埋めればよい。

---

## API/DB 変更点

### DB

変更なし。

### 必須の小改修：「単元なし」の問題だけを絞り込む

現状の `GET /v1/questions` は `unit_id` を渡さないと**その科目の全問題**（単元ありも含む）を返すため、「単元を持つ科目の直下（未分類）の問題だけ」を取得できない。ここだけ拡張する。

- 案A（推奨）: `unit_id` に未割り当てを表すセンチネルを追加（例: `?subject_id=…&unassigned_unit=true` で `unit_id IS NULL` を絞り込む）
- 案B: 専用パラメータを増やさず、フロントで `subject_id` の全件を取得してクライアント側で `unit == null` を振り分ける（件数が多いとページングと相性が悪い）

→ ページング整合のため案Aを推奨。`routers/questions.py::list_questions` にクエリ分岐を1つ足すだけ。

### 任意の改修：フォルダの件数バッジ

`SubjectResponse` / `UnitResponse` は現在 `id`・`name` のみで件数を持たない。バッジを出すには件数が要る。

- 案1（軽量・推奨）: 件数集計エンドポイントを1本追加（例: `GET /v1/browse/counts` が `{subject_id: 件数}` と `{unit_id: 件数}` を返す）。一覧APIのレスポンス形は変えない
- 案2: `SubjectResponse`/`UnitResponse` に `question_count` を追加。ただし他画面のレスポンスにも影響し、毎回集計コストがかかる

→ バッジは付加価値なので、まずは件数なしで出し、必要になったら案1で足すのがよい。

## 実装ステップ

1. `GET /v1/questions` に「単元なし」絞り込み（案A）を追加 → `openapi.json` 再生成 → `npm run generate`
2. フロント: `/browse` ルートページ（科目フォルダ一覧）
3. 科目ページ（単元フォルダ一覧＋未分類）／単元ページ（問題一覧）＋パンくず
4. 問題行から既存の問題詳細への遷移を接続
5. （任意）件数バッジ用の集計API（案1）と各フォルダへの表示
6. ナビゲーションに「フォルダ」への入口を追加

## 苦手問題ページとの住み分け（重要）

このページは既存の[苦手問題一覧](../design/screens/mistake-list.md)と一見かぶるが、**軸も対象データも別**であり、両立させる。混同すると片方が中途半端になる。

| | 苦手問題ページ | フォルダ・ブラウズ |
|---|---|---|
| 対象データ | `mistake_note` **がある問題だけ**（間違えた問題） | 登録した**全問題**（未整理・克服済みも含む） |
| 並べる軸 | **状態**（苦手中 / 克服済み） | **場所**（科目 → 単元の階層） |
| 主な操作 | 「復習する」「克服済みにする」 | たどる・詳細を開く |
| ユーザーの動機 | 「いま克服すべきはどれ？」 | 「あの問題どこだっけ／単元を棚卸ししたい」 |

とくに[クイック保存](./quick-save.md)で溜めた「ノートのない問題」は苦手問題ページには出ないが、このフォルダには出る——この差がフォルダを別に持つ根拠になる。

**方針（A案）:** 「状態で追う（苦手ページ）」と「場所で探す（フォルダ）」を重ねない。**苦手問題ページには科目・単元フィルタを足さず状態軸に専念させ**、科目・単元で探す導線はこのフォルダに寄せる。これにより2画面の守備範囲が交わらず、既存画面・API・DBスキーマの変更なし（本ページの新規追加と `GET /questions` の「単元なし」フィルタ追加のみ）で成立する。

## 検討事項・トレードオフ

- **差別化度は中**（ファイラ自体は定番UX）。ただし「間違いノートを科目・単元で棚卸しして振り返る」導線は既存の復習キューにない価値で、登録データの再利用性が上がる
- **空フォルダの扱い**: 問題が0件の科目・単元も表示するか。プリセットで大量の科目・単元を入れた場合、空フォルダだらけになりうる → 「空を隠す」トグル、または件数0を薄く表示、を検討
- **並び順**: 科目・単元は登録順/名前順どちらを既定にするか。プリセット由来なら `sort_order` を尊重したいが、現状の `subjects`/`units` に並び順カラムはない（プリセット機能側の設計と合わせて判断）
- **問題一覧の情報量**: フォルダ内の各問題に間違いノートの状態（active/mastered・間違い回数）をどこまで出すか。出すなら問題一覧に mistake_note の情報を載せる拡張が要る（別途検討）
- **クイック保存との相性**: [クイック保存](./quick-save.md)で未整理のまま溜めた問題も、このフォルダから科目・単元単位でまとめて見直せると整理導線として噛み合う
